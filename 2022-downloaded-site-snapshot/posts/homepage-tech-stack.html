Homepage Tech Stack

Over engineered? Yes and no.

My homepage is distributed between two services, one of which renders a universal single page application, and the other of which serves a GraphQL api wrapping a PostgreSQL database. (You can find the source code here.)

Was this overkill given the task at hand? Probably. Hosting static pages on github.io using Jekyll would have:

  • been much quicker and easier to develop.
  • been much quicker and easier to deploy.
  • been much easier to maintain.
  • totally worked fine.

Do I regret it? No. I wanted to build my homepage using some of the technologies I hoped to actually use at a future job. As a result, I have learned a lot about the strengths and weaknesses of my toolchain (and invariably those parts of other toolchains that I can compare and contrast with parts of my own).

Outline of the tech stack

To start, here is an outline of the main tools used for each part of the site. Of course, this is just an outline, and is therefore not an exhaustive list. For more detail, you can check out the entire source code for the site.

  • Frontend
    • Source
      • React: For building user interfaces.
      • Redux: To manage data flow.
      • Radium: To manage inline styles.
    • Development
      • Gulp: To manage common development tasks.
      • Eslint: To help me not make silly mistakes.
      • Webpack: To bundle javascript modules.
      • Babel: To enable modern javascript language features.

  • Backend

    • A Node.js service responsible for rendering the universal single page application. (Since it is universal, all of the tools mentioned for the frontend apply here as well.)

      • Source
        • Express: As a small convenience wrapper over node's native http module.
      • Deployment
        • NGINX: As a reverse proxy.
        • PM2: To manage node processes.
        • Upstart: To manage the node service as a linux service.

    • A Python service responsible for serving the GraphQL api that wraps my database.

      • Source
        • Django: For a few different reasons. See below.
        • Graphene: For building the GraphQL api.
      • Deployment
        • NGINX: As a reverse proxy.
        • Gunicorn: To serve the Django application.
        • PostgreSQL: To store my data!

Why Django? Didn't you say micro-service?

I agree, a Django project does not constitute a micro-service. I used Django anyways because I wanted to get my site up sooner rather than later, while still tackling the smallest possible case of a distributed architecture (just two services).

This way, I could focus on some of the more basic principles from the complicated art of managing a distributed architecture and less on trying to masterfully compose all of the services. It also helped me not get overwhelmed by a deep dive into the world of micro-services. Besides, I can (and will) further granularize this service later on.

Right now, I use Django (rather than something more lightweight such as Flask) mostly because Django can easily ship with an automatically generated website administration interface (I am essentially using it as a CMS). From the Django tutorial:

Generating admin sites for your staff or clients to add, change, and delete content is tedious work that doesn’t require much creativity. For that reason, Django entirely automates creation of admin interfaces for models.

Django is also responsible for serving the GraphQL api that wraps my database, but if this were all I needed, I would use something more lightweight (like Flask).

Conclusion

Anyways, this is where things stand at the moment. I'll continue to improve the project over time to boost my skills as a developer and maintainer. Feel free to reply with questions or comments below!
